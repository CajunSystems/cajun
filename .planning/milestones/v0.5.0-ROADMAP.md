## Phase Details

### Phase 7: Error Handling & Recovery Patterns тЬЕ
**Goal:** Demonstrate `Effect.fail`, `mapError`, and recovery composing inside effect actors.

Plans:
- 7.1 Write `EffectErrorHandlingExample.java` тАФ validation actor with fail/recover/mapError
- 7.2 Write `EffectRetryExample.java` тАФ retry-with-backoff via Effect chain

**API findings:**
- `Effect.suspend(ThrowingSupplier<A>)` re-evaluates supplier on each retry тАФ same `Effect` object reusable across retries
- `attempt()` widens error type to `Throwable` тАФ test methods need `throws Throwable`
- `@SuppressWarnings("unchecked")` required when casting `Either.Left`/`Either.Right`
- `catchAll` recovery effect must match original error type тАФ enables recursive retry chains

---

### Phase 8: Stateful Actor + Effect Actor Composition тЬЕ
**Goal:** Show `StatefulHandler` and `EffectActorBuilder` working side-by-side via the ask pattern.

Plans:
- 8.1 Write `EffectStatefulCompositionExample.java`
- 8.2 Write `EffectAskPatternExample.java`

**Key deviations/findings:**
- `StatefulActor` journals messages BEFORE `handler.receive()` тАФ ALL message/state types MUST implement `Serializable`
- UUID-based actor IDs required to avoid cross-run journal accumulation (file journal is append-only)
- `system.statefulActorOf(handlerInstance, initialState)` required when handler has constructor args
- `EffectActorBuilder` actors don't expose `ActorContext` тАФ use embedded `Pid replyTo` for replies

---

### Phase 9: Multi-Stage Effect Pipeline тЬЕ
**Goal:** A multi-actor processing pipeline where each stage is an effect actor.

Plans:
- 9.1 Write `EffectPipelineExample.java` тАФ 4-stage pipeline with LogCapability
- 9.2 Write `EffectFanOutExample.java` тАФ fan-out to worker pool with round-robin

**API findings:**
- `EffectActorBuilder` pipeline wiring: build sink-first, capture downstream Pid in upstream lambda closure
- `record Batch(List<String> items)` wrapper required тАФ raw generic types break type inference
- `AtomicInteger cursor` works safely in dispatcher lambda (actor processes one batch at a time)

---

### Phase 10: Custom Domain Capabilities тЬЕ
**Goal:** Show users how to define and compose custom capabilities.

Plans:
- 10.1 Write `EffectCapabilityExample.java` тАФ ValidationCapability + MetricsCapability
- 10.2 Write `EffectTestableCapabilityExample.java` тАФ swappable handlers

**API findings:**
- `Effect.generate(ctx -> ..., handler)` requires `.widen()` on handler
- `CapabilityHandler.compose(h1, h2, h3)` accepts raw unwidened handlers
- `Effect.from(cap)` + `withCapabilityHandler(h.widen())` = handler injected at spawn time (testable/swappable)
- Hold reference to stateful handler before spawn; inspect after latch for assertions

## API Knowledge Base (carried to Milestone 3)

| Pattern | Detail |
|---------|--------|
| Retry | Recursive `catchAll` chain; `Effect.suspend(supplier)` re-evaluates on each attempt |
| Stateful journals | ALL message/state types in StatefulActor must implement `Serializable` |
| UUID actor IDs | Use in tests with StatefulHandler to avoid cross-run journal accumulation |
| Reply pattern | Embed `Pid replyTo` in request record тАФ EffectActorBuilder has no ActorContext |
| Pipeline wiring | Build sink-first; capture downstream Pid in upstream lambda closure |
| Batch wrapper | `record Batch(List<T> items)` тАФ raw generic types break EffectActorBuilder type inference |
| Effect.generate() | Requires `.widen()` on handler argument |
| CapabilityHandler.compose() | Accepts raw (non-widened) handlers |
| Testable capabilities | `Effect.from(cap)` + `.withCapabilityHandler(h.widen())` = swappable at spawn time |
